1. 运行概览

1.1 整体 Loading

定义： 所有机台组的加权平均利用率，反映整体产能紧张程度。

前端字段： analysis.rccp.overall_loading_pct

计算公式：

整体Loading = Σ(所有机台组需求小时) / Σ(所有机台组可用小时) × 100%

计算过程：

# Step 1: 汇总所有机台组的需求小时
overall_demand = Σ demand_hours[tg]  # 遍历所有tool_group

# Step 2: 汇总所有机台组的可用小时
overall_available = Σ available_hours[tg]

# Step 3: 计算整体利用率
overall_loading_pct = (overall_demand / overall_available) × 100

数据来源： | 参数 | API | 字段 | |——|—–|——| | demand_hours | /rccp/compute | loading_table[].demand_hours | | available_hours | /rccp/compute | loading_table[].available_hours |

业务含义： | Loading范围 | 状态 | 业务解读 | |————-|——|———-| | < 75% | 健康 | 产能充足，可接更多订单 | | 75-85% | 警告 | 产能紧张，需关注瓶颈 | | 85-100% | 临界 | 产能不足，需调整计划 | | > 100% | 过载 | 无法满足需求，必须削减 |

示例：

假设3个机台组:
- LITHO_01: demand=2500h, avail=500h
- ETCH_03: demand=2240h, avail=800h
- DEPO_02: demand=400h, avail=400h

整体Loading = (2500+2240+400)/(500+800+400) × 100%
            = 5140/1700 × 100%
            = 302%

1.2 可行性（feasible）

定义： 当前需求计划是否在现有产能约束下可执行。

前端字段： analysis.rccp.feasible

判断逻辑：

feasible = True  # 初始值

for tool_group in loading_table:
    if tool_group.loading_pct >= 100%:
        feasible = False  # 任意机台过载 → 不可行
        break

状态分类阈值：

loading_pct >= 100%  → status = "overload"  → feasible = False
loading_pct >= 85%   → status = "critical"  → feasible仍True但需关注
loading_pct >= 75%   → status = "warning"   → feasible仍True但需预警
loading_pct < 75%    → status = "healthy"   → 正常

数据来源： | 参数 | API | 计算位置 | |——|—–|———-| | loading_table | /rccp/compute | 后端 rccp.py |

业务含义： - True: 计划可执行，产能足够 - False: 计划不可行，需削减或增加产能

1.3 主瓶颈（primary_bottleneck）

定义： 三维瓶颈评分最高的机台组，是当前最需要关注的瓶颈。

前端字段： analysis.bottleneck.primary_bottleneck

计算公式：

瓶颈评分 = Loading权重 × Loading分 + Queue权重 × Queue分 + Drift权重 × Drift分

权重配置：

weights = {
    "loading": 0.5,  # Loading维度权重50%
    "queue": 0.3,    # Queue维度权重30%
    "drift": 0.2,    # Drift维度权重20%
}

各维度评分计算：

Loading评分

loading_score = min(loading_pct, 100.0)  # 直接映射
# 97.9% loading → 97.9分

Queue评分（见第3章详细说明）

wait_hours = Kingman公式计算(utilization, service_rate, n_servers)
queue_score = 映射函数(wait_hours)  # 0-100分

Drift评分

drift_score = 斜率评分 + 加速度评分
# 基于历史4-8周loading趋势计算

主瓶颈判定：

bottlenecks.sort(key=composite_score, reverse=True)  # 按综合评分排序
primary_bottleneck = bottlenecks[0].tool_group_id if bottlenecks[0].severity in ("critical", "severe") else None

数据来源： | 参数 | API | 字段 | |——|—–|——| | loading_pct | /rccp/compute | loading_table[].loading_pct | | queue_score | /bottleneck/analyze | bottlenecks[].queue_score | | drift_score | /bottleneck/analyze | bottlenecks[].drift_score | | composite_score | /bottleneck/analyze | bottlenecks[].composite_score |

1.4 DES 结论

定义： 离散事件仿真验证结果，反映排队效应对可行性的影响。

前端字段： analysis.des.feasible

判断逻辑：

# DES仿真运行一周(168小时)
avg_wait_time = simulation_result.average_waiting_time
max_wait_time = simulation_result.max_waiting_time

# 判断标准
if avg_wait_time > threshold_hours:  # 阈值通常为24h
    feasible = False  # 等待时间过长 → 仿真不可行
else:
    feasible = True

数据来源： | 参数 | API | 字段 | |——|—–|——| | avg_wait_time | /des/run | simulation.avg_wait_hours | | feasible | /des/run | simulation.feasible |

业务含义： - 通过: 排队效应对Cycle Time影响可控 - 风险: 存在显著排队，需优化调度或增加产能

2. RCCP 热点

2.1 Tool Group ID

定义： 机台组唯一标识符，代表一组功能相同的设备。

前端字段： item.tool_group_id

数据来源： - 后端：route_master.tool_group_id（工艺路线定义） - 前端：loading_table[].tool_group_id

业务含义： - 如 LITHO_01 代表光刻区第一组设备 - 如 ETCH_03 代表刻蚀区第三组设备

2.2 Demand Hours（需求小时）

定义： 该机台组为满足计划需求所需的总小时数。

前端字段： item.demand_hours

计算公式：

demand_hours[tg] = Σ(产品产量 × 产品在该机台的TC)

计算过程：

# Step 1: 构建产能矩阵贡献
contribution_matrix = capacity_matrix × demand_plan
# 即每个产品在每个机台的需求贡献

# Step 2: 按机台聚合
for tg in tool_groups:
    demand_hours[tg] = Σ contribution_matrix[product][tg]

示例：

LITHO_01 需求计算:
- 28nm_DRAM_A: 1000片 × 2.5h/片 = 2500h
- 64L_NAND_B: 800片 × 2.0h/片 = 1600h
- 合计 = 2500 + 1600 = 4100h

数据来源： | 参数 | 来源 | 字段 | |——|——|——| | demand_plan | /rccp/compute请求体 | demand_plan | | capacity_matrix | /rccp/compute请求体 | capacity_matrix |

2.3 Available Hours（可用小时）

定义： 该机台组本周可用的总产能小时。

前端字段： item.available_hours

计算公式：

available_hours = 设备数 × 168h × OEE

计算过程：

# 从OEE表获取
for tg in tool_groups:
    n_machines = tool_groups[tg].n_machines
    oee = oee_table[tg].oee  # 综合设备效率
    
    available_hours[tg] = n_machines × 168 × oee

示例：

LITHO_01 可用产能:
- 设备数: 8台
- OEE: 75%
- 本周可用 = 8 × 168h × 0.75 = 1008h

数据来源： | 参数 | 来源 | 字段 | |——|——|——| | n_machines | Excel导入 | tool_groups.n_machines | | oee | Excel导入 | oee.oee | | available_hours | Excel导入或API | oee.available_hours |

2.4 Loading（利用率）

定义： 需求小时占可用小时的百分比，反映产能紧张程度。

前端字段： item.loading_pct

计算公式：

loading_pct = demand_hours / available_hours × 100%

计算过程：

loading_pct[tg] = demand_hours[tg] / available_hours[tg] × 100

示例：

LITHO_01 Loading:
loading_pct = 4100h / 500h × 100% = 820%

状态分类：

loading >= 100%  → overload  (红色)
loading >= 85%   → critical  (橙色)
loading >= 75%   → warning   (黄色)
loading < 75%    → healthy   (绿色)

2.5 Gap Hours（产能缺口）

定义： 需求超出可用产能的小时数，正值表示缺口。

前端字段： item.gap_hours

计算公式：

gap_hours = demand_hours - available_hours

示例：

LITHO_01 缺口:
gap_hours = 4100h - 500h = 3600h

业务含义： - gap > 0: 产能不足，缺口小时数 - gap ≤ 0: 产能充足

2.6 Status（状态）

定义： 机台组的产能健康状态。

前端字段： item.status

分类阈值：

def classify_status(loading_pct):
    if loading_pct >= 100%: return "overload"
    elif loading_pct >= 85%: return "critical"
    elif loading_pct >= 75%: return "warning"
    else: return "healthy"

2.7 排序逻辑

前端代码：

const topHotspots = analysis?.rccp?.loading_table?.slice(0, 12) || [];

排序规则： - 后端按 loading_pct 降序排序 - 前端只显示前12条（最热的机台组）

3. 瓶颈评分

3.1 Loading Score

定义： 基于利用率的瓶颈评分。

前端字段： item.loading_pct（同时用于显示和评分）

计算公式：

loading_score = min(loading_pct, 100.0)

评分映射：

97.9% loading → 97.9分
120% loading → 100分（上限）

3.2 Queue Score

定义： 基于排队等待时间的瓶颈评分，反映Cycle Time恶化程度。

前端字段： item.queue_score

计算公式：

Step 1: 计算等待时间（Kingman公式）

单服务器（G/G/1）：

Wq = (ρ/(1-ρ)) × ((Ca² + Cs²)/2) × (1/μ)

多服务器（G/G/c）Sakasegawa近似：

Wq = ρ^(√(2(c+1))) / (c × (1-ρ)) × ((Ca² + Cs²)/2) × (1/μ)

变量说明： | 变量 | 含义 | 默认值 | |——|——|——–| | ρ | 利用率 | loading_pct/100 | | μ | 服务率(wph) | 1.0（可配置） | | c | 设备数 | 1（可配置） | | Ca² | 到达CV² | 1.0（泊松到达） | | Cs² | 服务CV² | 0.5（半导体典型值） |

Step 2: 映射评分

def queue_score_from_wait(wait_hours):
    if wait_hours < 1:    return wait_hours × 20        # 0-20分
    elif wait_hours < 4:  return 20 + (wait-1) × 10     # 20-50分
    elif wait_hours < 12: return 50 + (wait-4) × 3.75   # 50-80分
    else:                 return min(80+(wait-12), 100) # 80-100分

示例：

LITHO_01 Queue评分:
输入参数:
- loading = 97.9% → ρ = 0.979
- service_rate = 50 wph → μ = 50
- n_servers = 8台 → c = 8

计算:
power = √(2×(8+1)) = 4.24
wait = 0.979^4.24 / (8×0.021) × 1.5/2 × 1/50
     ≈ 0.08h

评分:
wait = 0.08h < 1h → queue_score = 0.08 × 20 = 1.6分

数据来源： | 参数 | 来源 | 说明 | |——|——|——| | loading_pct | RCCP计算 | 已有 | | service_rate | 配置传入 | 默认1.0，可配置实际wph | | n_servers | 配置传入 | 默认1，可配置实际设备数 |

3.3 Expected Wait Hours

定义： Kingman公式计算的平均排队等待时间。

前端字段： item.expected_wait_hours

计算公式： 同Queue Score的Step 1

业务含义： - wait < 4h: Cycle Time影响可控 - wait 4-12h: 需关注调度优化 - wait > 12h: Cycle Time严重恶化

3.4 Drift Score

定义： 基于历史利用率趋势的瓶颈评分，反映问题恶化速度。

前端字段： item.drift_score

计算公式：

drift_score = slope_score + acceleration_score

计算过程：

Step 1: 计算线性斜率

# 历史loading序列: [88, 90, 92, 95, 97.9]
x = [0, 1, 2, 3, 4]  # 周序号
slope = polyfit(x, loading_history, 1)[0]  # 线性回归斜率
# slope ≈ 2.5 pp/周

Step 2: 计算加速度

# 前半段斜率
s1 = polyfit([0,1,2], [88,90,92], 1)[0] = 2.0

# 后半段斜率
s2 = polyfit([0,1,2], [95,97.9,?], 1)[0] = 3.0

# 加速度
accel = s2 - s1 = 1.0 pp/周²

Step 3: 评分映射

slope_score = min(max(slope × 10, 0), 50)   # 斜率评分
accel_score = min(max(accel × 25, 0), 50)   # 加速度评分
drift_score = slope_score + accel_score

示例：

LITHO_01 Drift评分:
历史4周: [88, 90, 92, 95] → 第5周97.9

斜率: 2.5 pp/周 → slope_score = 2.5×10 = 25分
加速度: 后半段加速 → accel_score = 12.5分
drift_score = 25 + 12.5 = 37.5分

数据来源： | 参数 | API | 字段 | |——|—–|——| | historical_loading | /bottleneck/analyze请求体 | historical_loading |

3.5 Composite Score（综合评分）

定义： 三维瓶颈综合评分，用于排名和预警。

前端字段： item.composite_score

计算公式：

composite_score = 0.5 × loading_score + 0.3 × queue_score + 0.2 × drift_score

示例：

LITHO_01 综合评分:
loading_score = 97.9
queue_score = 1.6
drift_score = 37.5

composite = 0.5×97.9 + 0.3×1.6 + 0.2×37.5
          = 48.95 + 0.48 + 7.5
          = 56.93分

3.6 Rank（排名）

定义： 按综合评分排序后的名次。

前端字段： item.rank

排序逻辑：

bottlenecks.sort(key=composite_score, reverse=True)
for i, b in enumerate(bottlenecks, 1):
    b.rank = i

3.7 Severity（严重程度）

定义： 瓶颈严重等级分类。

前端字段： item.severity

分类阈值：

def severity_from_score(score):
    if score >= 80: return "critical"   # 严重瓶颈
    elif score >= 60: return "severe"    # 较严重
    elif score >= 40: return "moderate"  # 中等
    else: return "minor"                 # 轻微

4. LP 优化器

4.1 Status

定义： LP求解器返回状态。

前端字段： analysis.lp.status

状态类型： | 状态 | 含义 | 说明 | |——|——|——| | optimal | 最优解找到 | 成功求解 | | infeasible | 问题不可行 | 无满足约束的解 | | timeout | 超时 | 未在时限内求解 | | heuristic | 启发式解 | CBC失败，用贪心算法 | | error | 求解错误 | solver异常 |

4.2 Objective Value（目标值）

定义： LP目标函数值（最大化利润时的总利润）。

前端字段： analysis.lp.objective_value

计算公式：

objective_value = Σ(unit_profit[product] × optimal_plan[product])

示例：

目标值计算:
- 28nm_DRAM_A: 200片 × ¥150/片 = ¥30,000
- 64L_NAND_B: 0片 × ¥80/片 = ¥0
- 目标值 = ¥30,000

4.3 Optimal Plan（最优产量分配）

定义： LP求解的最优产品产量分配。

前端字段： analysis.lp.optimal_plan

计算逻辑：

LP模型:
max Σ profit[i] × x[i]

约束:
Σ TC[i,j] × x[i] ≤ H[j]  ∀j  (产能约束)
x[i] ≥ demand_min[i]        (合同最低)
x[i] ≤ demand_max[i]        (市场上限)
x[i] ≥ 0

求解器： CBC (COIN-OR Branch and Cut)

4.4 Capacity Utilization

定义： LP最优解下各机台组的利用率。

前端字段： analysis.lp.capacity_utilization

计算公式：

utilization[tg] = Σ(TC[product][tg] × optimal_plan[product]) / available_hours[tg] × 100%

4.5 Binding Constraints

定义： 达到产能上限的机台组（紧约束）。

前端字段： analysis.lp.binding_constraints

判断逻辑：

binding = []
for tg in tool_groups:
    utilization = capacity_utilization[tg]
    if utilization >= 99.5%:
        binding.append(tg)  # 利用率接近100% → 紧约束

recommendations = []

7.1 Output Target Week（产出目标周）

前端字段： outputConfig.outputTargetWeek

数据来源： - 用户输入或 - datasetSummary.time_windows[0]

7.2 Output Targets（产出目标）

定义： 各产品本周的产出目标量。

前端字段： outputConfig.outputTargets

计算逻辑：

targets[product] = Math.round(demand_plan[product] × 0.8)  // 默认80%

数据来源： | 参数 | API | 字段 | |——|—–|——| | demand_plan | /data/demand_plan | demand_plan |

7.3 WIP Lot Data

定义： 当前在制Lot明细数据。

前端字段： wipData

必需字段： | 字段 | 含义 | 示例 | |——|——|——| | lot_id | Lot编号 | LOT001 | | product_id | 产品ID | 28nm_DRAM_A | | current_step_seq | 当前工序序号 | 60 | | wafer_count | 晶圆数 | 25 | | percent_complete | 完成度% | 85 |

工序序号计算：

stepsPerProduct = Math.round(n_routes / n_products)  // 约70步
current_step_seq = Math.floor(stepsPerProduct × stepPct)
// 85%完成度 → current_step = 70 × 0.85 = 59

7.4 Total Predicted Output（总预测产出）

定义： WIP可贡献的本周总产出量。

前端字段： outputResult.total_predicted_output

计算逻辑：

predicted_output = Σ(near_complete_lots.wafer_count)
# near_complete: percent_complete >= threshold(80%)

7.5 Output Gap（产出缺口）

定义： 产出目标与WIP预测的差值。

前端字段： outputResult.output_gap

计算公式：

output_gap[product] = output_target[product] - predicted_output[product]

7.6 Overall WIP Share（WIP占比）

定义： WIP后续工序小时占总需求小时的百分比。

前端字段： outputResult.overall_wip_share_pct

计算公式：

overall_wip_share = Σ(WIP后续工序小时) / Σ(总需求小时) × 100%

示例：

若只有WIP（无新投入）:
WIP后续工序小时 = 总需求小时
overall_wip_share = 100%

7.7 Capacity Demand Breakdown

定义： 各机台组的产能需求分解。

前端字段： outputResult.capacity_demand

字段说明： | 字段 | 含义 | 计算公式 | |——|——|———-| | wip_remaining_hours | WIP后续工序小时 | Σ后续工序TC × wafers | | new_input_hours | 新投入小时 | 新投入 × 本周工序TC | | total_demand_hours | 总需求 | wip + new_input | | loading_pct | 利用率 | demand / available × 100 | | wip_share_pct | WIP占比 | wip / total × 100 |

7.8 Input Recommendations（投入建议）

定义： 为满足产出目标需要的新投入量。

前端字段： outputResult.input_recommendations

计算公式：

input_recommendation[product] = output_gap[product]
# 建议投入量 = 产出缺口

7.9 Completion Rate（完成度）

定义： 预测产出占目标的比例。

前端计算：

completion_rate = total_predicted_output / Σ(outputTargets) × 100%

8. What-if 分析

8.1 扰动类型

定义： 模拟的异常场景类型。

前端字段： whatIfConfig.perturbationType
