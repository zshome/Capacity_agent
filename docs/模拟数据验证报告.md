# Capacity Agent 模拟数据验证报告

## 验证目标

本次验证用于确认本轮优化后的算法口径是否能够支撑 R6 主生产计划下的投片需求审查、产能承诺预检和周计划冻结前评估。

验证重点不是单纯检查接口可调用，而是确认系统不会输出高估产能或掩盖不可行风险的计划。

## 模拟数据覆盖

| 验证场景 | 模拟设定 | 预期结论 |
| --- | --- | --- |
| 情景识别 | 相同 Path + Backup、不同 Path + Backup | 相同 Path 走标准产能；不同 Path + Backup 走分配模型 |
| WIP 口径 | 新计划 80 小时，可用 100 小时，WIP 后续负载 30 小时 | 不含 WIP 可行；含 WIP 后 loading=110%，不可行 |
| 多制程完成量 | 单台设备 100 小时，产品需 PHOTO+ETCH 两道工序，各 1 小时/wafer | 完整可承诺量只能是 50 wafers，不能误算为 100 wafers |
| 不同 Path + Backup 分配 | DRAM/NAND 共享 LITHO/ETCH，LITHO 存在 Backup | 输出未满足需求和瓶颈设备，不生成虚假满额计划 |
| Excel 导入 | 使用系统模板导入 5 个必填 Sheet + 复杂 Path 推荐 Sheet | 可生成复杂 Path 分配模型输入 payload |
| 内置复杂 Path Demo | `Complex Path + Backup Demo`，含 2 个产品、5 台单机台、5 个制程、12 条能力矩阵、2 条 Backup | 前端可直接选择并触发 `scenario_5` 与分配模型 |

## 验证结果

执行命令：

```bash
PYTHONPATH=capacity_agent python3 capacity_agent/validate_company_method_readiness.py
```

结果摘要：

| 用例 | 结果 | 关键输出 |
| --- | --- | --- |
| Scenario routing | PASS | `scenario_3` 使用标准产能，`scenario_5` 使用分配模型 |
| WIP-aware R6 RCCP | PASS | 含 WIP 后 loading 为 `110%`，计划不可行 |
| Completed wafer allocation | PASS | 完整完成量 `50 wafers`，未满足需求 `50 wafers` |
| Different path + backup allocation | PASS | DRAM 输出 `80`，NAND 输出 `48`，NAND 缺口 `32` |
| Excel complex path template | PASS | 模板导入后 `complex_path_ready=True` |
| Built-in complex path demo | PASS | 前端下拉可选，运行后识别为 `scenario_5`，显示复杂 Path 分配结果 |

## 审查结论

当前系统已经具备正式使用前的核心算法验证基础：

- R6 投片计划审查会同时考虑新计划负载和 WIP 后续负载。
- 不同 Path / Backup 场景会进入产品-制程-机台分配模型。
- 分配模型以“完整产品量”为输出口径，不再把多道制程加工量累加成产出。
- 当产能不足时，系统会输出 unmet demand 和瓶颈机台，而不是给出不可执行的满额计划。
- Excel 模板已经能够承载正式计算需要的基础资料和复杂 Path 扩展资料。
- 内置样例已补充复杂 Path + Backup 数据集，可直接用于前端联调和项目评审演示。

## 正式使用前条件

当前启发式分配结果适合用于预检、方案推演和项目评审演示。若要用于正式投片决策、产能承诺或周计划冻结，还需要满足以下条件：

1. 生产环境安装并启用 LP/CBC 或同等级求解器。
2. 公司真实数据需要补齐 `tool_master`、`process_master`、`tool_process_capability`、`backup_path`。
3. R6 月度需求、WIP Lot、OEE/可用小时需要建立固定刷新频率和版本留痕。
4. 需要用历史月份做回测，对比系统建议投片量、实际投片量、瓶颈设备利用率和实际达交情况。
