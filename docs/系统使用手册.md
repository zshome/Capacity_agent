## 1. 系统概述

### 1.1 系统定位

Capacity Agent 是一个**半导体制造产能规划智能系统**，解决以下核心问题：

| 问题类型 | 传统方式 | Capacity Agent 解决方案 |
|----------|----------|------------------------|
| 产能瓶颈识别 | 人工计算、主观判断 | 三维瓶颈评分（Loading + Queue + Drift） |
| 产品 Mix 优化 | Excel 手动调整 | LP 自动优化（最大化利润/产出） |
| WIP 产出预测 | 基于总CT估算 | 工序级TC精确计算 |
| What-if 分析 | 多次重新计算 | 一键扰动重算 |
| 跨视角规划 | 单一投入视角 | Input + Output 双视角 |

### 1.2 技术架构

## 3. 功能模块详解

### 3.1 RCCP（粗略产能规划）

#### 功能定位

**问题场景：**
> "本周计划投入 1000 片 28nm DRAM，产能够不够？"

#### 计算逻辑

```
Step 1: 构建产能矩阵
  capacity_matrix[product][tool_group] = Σ (run_time_hr / batch_size)
  
Step 2: 计算需求小时
  demand_hours[tool_group] = Σ demand_plan[product] × capacity_matrix[product][tool_group]
  
Step 3: 计算利用率
  loading_pct = demand_hours / available_hours × 100%
  
Step 4: 判断可行性
  feasible = 所有 loading_pct < 100%
```

#### 输出结果

| 指标 | 含义 | 业务意义 |
|------|------|----------|
| loading_pct | 机台组利用率 | >100% = 过载瓶颈 |
| feasible | 整体可行性 | False → 需调整计划 |
| gap_hours | 产能缺口 | 需增加多少小时产能 |

#### 使用示例

```javascript
// API 调用
POST /rccp/compute
{
  "demand_plan": {"28nm_DRAM_A": 1000, "64L_NAND_B": 800},
  "capacity_matrix": {"28nm_DRAM_A": {"LITHO_01": 2.5}},
  "available_hours": {"LITHO_01": 500}
}

// 返回结果
{
  "loading_table": [{"tool_group_id": "LITHO_01", "loading_pct": 820, "status": "overload"}],
  "feasible": false,
  "overall_loading_pct": 733
}
```

### 3.2 瓶颈分析（Bottleneck Analyzer）

#### 功能定位

**问题场景：**
> "哪个机台组是真正的瓶颈？光刻区利用率高但不一定是瓶颈？"

#### 计算逻辑（三维评分）

```
瓶颈评分 = Loading权重 × Loading评分 + Queue权重 × Queue评分 + Drift权重 × Drift评分

Loading评分:
  - >100%: 100分（过载）
  - 85-100%: 80分（临界）
  - 75-85%: 60分（警告）
  - <75%: 30分（健康）

Queue评分（历史等待时间）:
  - avg_wait > 48h: 100分
  - avg_wait 24-48h: 60分
  - avg_wait < 24h: 30分

Drift评分（利用率漂移）:
  - 周波动 > 20%: 80分（不稳定）
  - 周波动 < 5%: 30分（稳定）
```

#### 输出结果

```
瓶颈排名:
┌────────────┬─────────┬──────────┬──────────┬────────┐
│ 机台组     │ Loading │ Queue(h) │ 漂移     │ 总评分 │
├────────────┼─────────┼──────────┼──────────┼────────┤
│ LITHO_01   │ 120%    │ 72h      │ 高波动   │ 95     │
│ ETCH_03    │ 95%     │ 48h      │ 中波动   │ 75     │
│ DEPO_02    │ 80%     │ 12h      │ 低波动   │ 45     │
└────────────┴─────────┴──────────┴──────────┴────────┘
```

### 3.3 LP Optimizer（线性规划优化）

#### 功能定位

**问题场景：**
> "产能不够，应该优先生产哪些产品？如何调整产品 Mix？"

#### 计算逻辑

```
目标函数（最大化利润）:
  max Σ unit_profit[product] × x[product]

约束条件:
  Σ capacity_matrix[product][tool_group] × x[product] ≤ available_hours[tool_group]
  x[product] ≥ demand_min[product]  （合同最低量）
  x[product] ≤ demand_max[product]  （市场需求上限）

求解器: CBC (开源) 或 GLPK
```

#### 输出结果

```
LP优化结果:
┌─────────────┬─────────────┬────────────┐
│ 产品        │ 原计划      │ LP最优分配 │
├─────────────┼─────────────┼────────────┤
│ 28nm_DRAM_A │ 1000片      │ 200片      │ ← 削减80%
│ 64L_NAND_B  │ 800片       │ 0片        │ ← 完全放弃
│ 128L_NAND   │ 600片       │ 150片      │ ← 削减75%
└─────────────┴─────────────┴────────────┘

目标值（总利润）: ¥30,000（较原计划 ¥162,000 大幅下降）
瓶颈机台: LITHO_01（利用率100%）
```

#### 业务解读

```
LP 建议解读:
├── 产能严重不足，无法满足全部需求
├── 优先生产高利润产品（28nm_DRAM_A: ¥150/片）
├── 放弃低利润产品（64L_NAND_B: ¥80/片）
├── 建议措施: 增加 LITHO_01 产能 或 降低需求
```

### 3.4 DES 仿真验证

#### 功能定位

**问题场景：**
> "RCCP 显示光刻区利用率 120%，但实际排队情况如何？仿真验证一下。"

#### 计算逻辑

```
仿真参数:
├── 机台组: LITHO_01 (8台机)
├── 服务率: 50 wph (每小时处理50片)
├── 到达率: 60 wph (每小时到达60片需求)
├── 服务CV: 0.5 (服务时间波动)
├── 仿真时长: 168小时 (一周)

仿真结果:
├── 平均利用率: 85%
├── P95等待时间: 48小时
├── 平均周期时间: 72小时
├── 仿真可行性: False（等待过长）
```

#### 与 RCCP 对比

| 方法 | 优势 | 局限 |
|------|------|------|
| RCCP | 快速、静态 | 无法反映排队效应 |
| DES | 动态、精确 | 计算耗时 |

### 3.5 Output 视角（产出规划）

#### 功能定位

**问题场景：**
> "下周要产出 1000 片，当前 WIP 能贡献多少？需要新投入多少？"

#### 计算逻辑

**与传统 Input 视角的区别:**

```
Input 视角（传统）:
  需求 = 新投入量 × 总工时
  
Output 视角（本系统）:
  需求 = WIP后续工序小时 + 新投入本周工序小时
```

**工序级 TC 计算:**

```
Step 1: 计算每个 Lot 的后续工序
  current_step_seq = 60 (当前在第60步)
  remaining_steps = route[step_seq > 60] (后续工序)
  
Step 2: 计算后续工序总工时
  remaining_hours = Σ run_time_hr × wafer_count
  = (0.3 + 0.4 + 0.2) × 25 = 22.5h
  
Step 3: 预测完成周
  remaining_days = remaining_hours / 24 = 0.94天
  completion_week = current_week + 1周
```

#### 输出结果

```
Output RCCP 结果:
┌────────────┬───────────┬─────────────┬──────────┐
│ 产品       │ 产出目标  │ WIP可贡献   │ 缺口     │
├────────────┼───────────┼─────────────┼──────────┤
│ 28nm_DRAM  │ 1000片    │ 25片        │ 975片    │
│ 64L_NAND   │ 800片     │ 0片         │ 800片    │
└────────────┴───────────┴─────────────┴──────────┘

投入建议:
├── 28nm_DRAM: 需新投入 975 片
├── 投入时间: 产出周 - CT（约14周前）
├── 风险: 当前 WIP 完成度过低，本周产出有限
```

### 3.6 What-if 分析

#### 功能定位

**问题场景：**
> "如果光刻机停机 24 小时，产能影响多大？"

#### 计算逻辑

```
Step 1: 保存基线 RCCP 结果
  baseline_loading = {LITHO_01: 120%, ETCH_03: 95%}
  
Step 2: 应用扰动
  perturbation = {tool_group_id: "LITHO_01", hours_lost: 24}
  
Step 3: 重新计算 RCCP
  new_available_hours["LITHO_01"] = 500 - 24 = 476h
  new_loading = RCCP.compute(demand, capacity, new_hours)
  
Step 4: 计算差异
  delta_loading = new_loading - baseline_loading
  status_change = baseline_status != new_status
```

#### 输出结果

```
What-if 分析结果:
┌────────────┬─────────┬─────────┬────────┬─────────┐
│ 机台组     │ 基线    │ 扰动后  │ Delta  │ 状态变化│
├────────────┼─────────┼─────────┼────────┼─────────┤
│ LITHO_01   │ 120%    │ 125%    │ +5pp   │ Yes     │
│ ETCH_03    │ 95%     │ 95%     │ 0pp    │ No      │
└────────────┴─────────┴─────────┴────────┴─────────┘

结论: 光刻机停机24h → 过载加剧5%，需紧急调整计划
```

## 4. 应用场景

### 4.1 场景一：周产能规划

```
问题描述:
  每周一制定本周生产计划，需要快速判断产能是否足够

使用流程:
  1. 导入本周需求计划（demand_plan）
  2. 点击"运行完整分析"
  3. 查看 RCCP 结果 → feasible: True/False
  4. 若不可行 → 查看 LP 优化建议
  5. 调整需求计划 → 重新分析

预期结果:
  ├─ 明确瓶颈机台组
  ├─ 知道产能缺口量
  ├─ 获得最优产品 Mix 建议
```

### 4.2 场景二：产能瓶颈排查

```
问题描述:
  某产品出货延迟，需要定位瓶颈原因

使用流程:
  1. 运行完整分析
  2. 查看"瓶颈评分"面板
  3. 查看瓶颈机台的 Loading + Queue + Drift
  4. 分析瓶颈成因:
     ├─ Loading高 → 产能不足
     ├─ Queue长 → 调度问题
     ├─ Drift大 → 不稳定
  5. 运行 DES 仿真验证

预期结果:
  ├─ 三维评分识别真正瓶颈
  ├─ 区分产能瓶颈 vs 调度瓶颈
  ├─ 仿真验证排队情况
```

### 4.3 场景三：紧急情况应对

```
问题描述:
  光刻机突发故障停机24小时，如何调整计划

使用流程:
  1. 切换到 What-if 面板
  2. 选择扰动类型: "机台停机"
  3. 设置: tool_group_id = LITHO_01, hours_lost = 24
  4. 点击"运行 What-if"
  5. 查看影响范围和程度
  6. 根据 LP 建议调整计划

预期结果:
  ├─ 快速量化停机影响
  ├─ 知道哪些产品受影响最大
  ├─ 获得调整后的最优计划
```

### 4.4 场景四：WIP 产出预测

```
问题描述:
  下周客户要 1000 片，当前 WIP 能贡献多少

使用流程:
  1. 导入 WIP Lot 数据（MES导出）
  2. 启用 Output 视角
  3. 设置产出目标: {"产品": 1000片}
  4. 点击"运行 Output RCCP"
  5. 查看 WIP 可贡献量 + 产出缺口
  6. 查看投入建议（投入量 + 投入时间）

预期结果:
  ├─ 精确预测 WIP 本周产出
  ├─ 计算新投入需求
  ├─ 反推投入时间窗口
```

#### RCCP 验证

| 指标 | 系统计算 | 手动验证 | 误差容忍 |
|------|----------|----------|----------|
| Loading% | Σ(demand × TC) / available | Excel 公式计算 | < 5% |
| 可行性 | 全部 Loading < 100% | 人工判断 | 一致 |
| 瓶颈机台 | Top 3 loading | 经验判断 | 排名一致 |

#### LP 验证

| 指标 | 系统计算 | 实际数据 | 误差容忍 |
|------|----------|----------|----------|
| 最优产量 | LP求解 | 实际生产量 | 需解释差异 |
| 总利润 | Σ(profit × qty) | 财务核算 | < 10% |
| 瓶颈机台 | Binding constraints | 实际瓶颈 | 一致 |

#### Output 验证

| 指标 | 系统预测 | 实际数据 | 误差容忍 |
|------|----------|----------|----------|
| WIP 产出 | 预测值 | MES 实际产出 | < 10% |
| 完成周 | 预测周 | 实际完成周 | ±1周 |
| 后续工时 | Σ run_time | MES 工时记录 | < 5% |

### 5.3 验证脚本示例

```python
# RCCP 计算验证脚本
import pandas as pd
from engines.rccp import compute_rccp

# 加载厂内数据
route = pd.read_excel("厂内数据.xlsx", sheet_name="route_master")
tool_groups = pd.read_excel("厂内数据.xlsx", sheet_name="tool_groups")
demand = pd.read_excel("厂内数据.xlsx", sheet_name="demand_plan")

# 系统计算
result = compute_rccp(demand, route, tool_groups)

# 手动验证（Excel 公式）
# 例如: LITHO_01 Loading = (1000×2.5 + 800×2.0) / 500 = 820%

print(f"系统计算 Loading: {result['loading_table'][0]['loading_pct']}%")
print(f"手动验证 Loading: 820%")
print(f"误差: {abs(result['loading_table'][0]['loading_pct'] - 820)}%")
```

### 5.4 常见验证问题

| 问题 | 可能原因 | 解决方案 |
|------|----------|----------|
| Loading 误差 > 5% | batch_size 配置错误 | 核查 route_master.batch_size |
| 瓶颈排名不一致 | OEE 数据时效性 | 使用最近一周 OEE |
| LP 结果与实际差距大 | 利润参数不准确 | 更新 unit_profit |
| Output 预测偏差大 |工序TC 不准确 | 核查 run_time_hr 实测值 |
